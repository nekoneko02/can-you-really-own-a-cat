# AI駆動開発におけるIssue運用方針

## 基本原則

Issue は **AIに渡す実行可能な設計コンテキスト** として運用する。

* **人間**: Why（目的・価値・制約）を決める
* **AI**: How（設計・分解・実装）を担当する

---

## 1. Issue 構造（例）

```
[Intent] ビジネス意図
  └─ [Unit] 実装単位
       └─ [Unit] サブ Unit（必要な場合のみ）
```

### ルール

1. **Intent に対応する Issue は必ず作る**
2. **Unit に対応する Issue は必ず作る**
3. **Unit が大きい・複雑な場合**:
   - サブ Unit Issue に分割する、または
   - Unit 自体を再設計して小さくする
4. **親 Issue へのリンクは必須**
5. Issue単体で作業ができるように必要な情報を記載すること（参照が必要なファイルパスも含む）。
   1. ただし、作業者は、必要に応じて、why部分は親Issueを通して確認すること（つまり、サブIssueにwhy部分を網羅的に記載する必要はない）。

---

## 2. Issueの役割

### 必須項目（共通）

- 背景・目的
- 概要
- 参考情報（ファイルパス、もしくはIssueへのリンク）

### Intent Issue

* なぜこの機能・変更を行うのか
* 解きたい問題・価値仮説
* 全体の非機能要件・制約
* Unit を束ねる親

### Unit Issue

* **AI に渡す基本作業単位**
* 設計〜実装〜テストまでを含む
* **単体で実装可能** であること

---

## 3. Issue 作成フロー

1. **AI が Issue 内容を `docs/work/` 配下で整理**
2. **人間がレビュー・承認**
3. **承認後、指示に従い GitHub Issue を作成**

---

## まとめ（一文で）

> **Intent/Unit に対応する Issue は必ず作り、単体で実装可能な情報を含める。**
