# AI-Driven Development Lifecycle (AI-DLC) とは

## AI-Driven Development Lifecycle (AI-DLC) の総合まとめ

### I. AI-DLCの概要と基本原則

AI-DLC（AI-Driven Development Life Cycle：AI駆動開発ライフサイクル）は、「AIを開発プロセスの中心的な協力者およびチームメイトとして位置づけ、ソフトウェア開発ライフサイクル全体に組み込む手法」です。これは、従来の開発手法（SDLCやアジャイル）がAIのスピードや能力に対応できないため、開発プロセスをAIネイティブなものへと再構築することを目指しています。

#### 1. 主要な特徴と原則

* **AIが実行し、人間が監視する**：開発ワークフローはAIが中心となり、計画の立案と実行を行います。人間の仕事は、計画や実行の検査、**「判断・意思決定」に注力される**ことになります。AIは体系的に詳細な作業計画を作成し、重要な決定は人間に委ねます。
* **会話の方向性の逆転**：AI-DLCでは、人間がAIとの会話を開始するのではなく、**AIが人間との会話を開始し、指示する**という根本的なシフトを導入しています。AIは高レベルの意図（Intent）を実行可能なタスクに分解し、レコメンデーションを生成し、トレードオフを提案します。
* **迅速なサイクル**：従来の開発手法が週や月単位の長いイテレーションのために構築されていたのに対し、AI-DLCは**時間または日単位**で測定される迅速なサイクル（Bolt）を前提としています。
* **高度なシステムへの対応**：AI-DLCは、継続的な機能適応性、高いアーキテクチャの複雑性、多くのトレードオフ管理、スケーラビリティ、統合、およびカスタマイズ要件を必要とする**複雑なシステムを構築する**ことを目的としています。
* **協業とリアルタイムでの問題解決**：AIがルーティンタスクを処理する一方で、チームは**リアルタイムでの問題解決、創造的思考、迅速な意思決定**のために、コラボレーションしやすい環境で協力します。ステークホルダーとの物理的な空間を共有することが推奨されます。

### II. AI-DLCの詳細：コアフレームワーク

AI-DLCのコアフレームワークは、「成果物 (Artefacts)」、「フェーズと儀式 (Phases & Rituals)」、「ワークフロー (The Workflow)」の3つの主要な要素で構成されます。

#### 1. フェーズと儀式

AI-DLCは大きく3つのフェーズで進行します。

| フェーズ | 目的 | 主な儀式 | AIと人間の役割 |
| :--- | :--- | :--- | :--- |
| **Inception** (インセプション) | ビジネス意図（Intent）を捕捉し、開発用のユニット（Unit）に変換する。 | **Mob Elaboration**（モブ・エラボレーション）。 | AIが意図をユーザーストーリー、受け入れ基準、ユニットに分解する初期案を提案し、人間（モブ）が検証・修正する。 |
| **Construction** (構築) | 定義されたユニットを、テスト済みで運用可能なDeployment Unitsに変換する反復的な実行。 | **Mob Construction**（モブ・コンストラクション）、Mob Programming、Mob Testing。 | AIがDomain DesignからLogical Design、コード生成、自動テスト実施を主導し、開発者が重要な決定と検証を行う。 |
| **Operation** (運用) | システムのデプロイメント、可観測性、メンテナンスに焦点を当てる。 | 特になし（継続的な監視） | AIがテレメトリーデータを能動的に分析し、異常を検出、SLA違反を予測し、実行可能な推奨事項を提案する。人間は検証者として機能する。 |

#### 2. 主要な成果物（Artefacts）

* **Intent（インテント）**: 達成すべき目的のハイレベルな記述であり、AI駆動のタスク分解の始点となる。
* **Unit（ユニット）**: Intentから派生した、まとまりがあり、自己完結型の作業要素。DDDのサブドメインやScrumのエピックに類似し、並行開発を可能にする。
* **Bolt（ボルト）**: AI-DLCにおける最小のイテレーション単位であり、**数時間または数日**で測定されるビルド・検証サイクルを持つ（Scrumのスプリントに類似）。
* **Domain Design/Logical Design**: AIがDDDの原則を用いてコアビジネスロジックをモデル化し（Domain Design）、NFRsを適用してアーキテクチャ設計パターンを組み込む（Logical Design）。
* **Deployment Units**: 機能テスト、セキュリティ、NFR、その他のリスクについて厳密にテストされた、パッケージ化された実行可能コード、構成、およびインフラコンポーネント。

#### 3. ワークフローの特徴

* **計画と人間の監視**: AIはまずLevel 1 Planを生成し、人間はそれをレビュー、検証、洗練します。
* **損失関数としての人間**: 各ステップは戦略的な意思決定ポイントであり、人間の監視は**「損失関数」**のように機能します。これにより、エラーが下流で雪だるま式に拡大する前に、早期に捕捉し修正することが保証されます。
* **設計技術の統合**: アジャイル手法とは異なり、AI-DLCは設計技術（例：**ドメイン駆動設計/DDD**）をその不可欠なコアとして組み込みます。

### III. AI-DLCの事例

AI-DLCは、AWSが提供するワークショップ型プログラム**「AI-DLC Unicorn Gym」**を通じて、複数の企業で実践・検証されています。

| 企業名 | 適用テーマ/目的 | 実施結果/学び |
| :--- | :--- | :--- |
| **東京海上日動システムズ** | **抜本的な生産性向上（10倍、20倍）**。既存システム改修と新規システム構築の効率化検証。 | **1.5日で4チームすべてが動作するシステムの初期版を完成**。従来数週間〜数ヶ月かかる作業が劇的に短縮。**DDDのような高度な設計パターン**をAIと協力することでスムーズに導入できることを確認。 |
| **ディップ株式会社** | スポットバイトルにおける使用者都合の早退時の運用フロー改善施策（労働法が関わる重要施策）。 | **法律観点を踏まえたユーザーストーリーの整理**が行え、企画段階で精度高く検討できるメリットを享受。施策全体でAIを活用した開発フローを体験。リアーキテクチャプロジェクトに学んだ手法を早速適用中。 |
| **DMM.com** | プロセスを「AI」前提に作り変え、**開発全体のリードタイムを短縮**する。 | **最大で80%の工数削減**が見込める結果となった。これはAIによる効率化と、**モブプロによるリアルタイムな意思決定、工程間の待ち時間削減**が寄与した。全4チームがコード生成から動作確認まで完了。 |
| **LINEヤフー株式会社** | 負荷試験環境の構築（Locust設定、API/DB初期化スクリプト作成）へのAI-DLC適用。 | AI-DLCのアウトプットを新規担当者の**キャッチアップ資料**として活用するなど、軽量にカスタマイズして取り入れている。 |

### IV. AI-DLCの課題と導入時の判断軸

AI-DLCの導入は抜本的な変革を伴うため、実践にあたってはいくつかの課題が認識されています。

#### 1. AI-DLCの課題や難しい点

* **同期的な時間確保の難しさ（最大の課題）**：AI-DLCは、ステークホルダーが**物理的な空間を共有**し、リアルタイムで問題解決や意思決定を行うための**同期的な時間**を確保することが前提とされていますが、これが最も多く挙げられた課題です。
* **既存システム統合の難しさ（Brown-Field）**：既存のDDDでないシステムに新しい機能を合流させる際、**ドメインとコードの機能マッピングのような追加のコンテキストをAIに大量に渡す必要**があり、AIコーディングの旨味が薄れてしまうという経験が報告されています。
* **人間の能力への依存**：AI中心の開発になったとしても、**結局のところAIのアウトプット品質は人間の能力に依存するところが大きい**という見解があります。即時判断を行う人間の能力の高さが求められます。
* **意思決定者のボトルネック化**: AIが単一プロセスを高速化する一方で、リアルタイムでの判断が求められる**意思決定者（PdM）がボトルネックになる**可能性があるという指摘があります。
* **技術的なスキルと知識の不足**：AI駆動開発を効果的に進めるためには、**まだまだ学ぶべきことが多い**という声があります。

#### 2. 導入時の判断軸（適しているプロジェクト）

AI-DLCは、特定の要件を持つプロジェクトに最大限の効果を発揮します。

| 導入が適している/効果的なプロジェクト | 導入が適していない/効果が薄いプロジェクト |
| :--- | :--- |
| **複雑なシステム**：継続的な機能適応性、高いアーキテクチャの複雑性、多くのトレードオフ管理、スケーラビリティ、統合、カスタマイズが求められるシステム。 | **非常にシンプルなシステム**：トレードオフ管理がほとんど必要なく、非開発者のペルソナによって開発できるシステムは、**ローコード/ノーコードアプローチ**の方が適している。 |
| **ドメイン駆動設計（DDD）の原則を適用したいプロジェクト**：AI-DLCはDDDとの相性が非常に良いため、DDDの概念を理解し、適用することでAIのポテンシャルを最大限に引き出せます。 | **DDDの原則から大きく逸脱した既存システム**：既存のドメイン知識やコンテキストの伝達が困難な場合、AIの活用効果が薄れる可能性があります。 |
| **抜本的な生産性向上を目標とするプロジェクト**：従来の延長線上の改善（10〜20%）ではなく、10倍、20倍といった抜本的な変革を目指す場合。 | **過度に小さな施策**：小さすぎるとAI-DLCの手法が過剰となり、効果が薄れる印象があります。 |
| **リアルタイムでの意思決定と共同作業が可能な体制が整っているプロジェクト**：モブ・エラボレーションなどの儀式を実施できる同期的な時間を確保できる体制が必要です。 | |
