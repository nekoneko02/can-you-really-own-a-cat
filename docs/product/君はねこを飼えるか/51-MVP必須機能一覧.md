## 1. オンボーディング機能

### 1-1. プレイ前情報入力 🔴

**対応する理由**: ユーザーの初期状態測定（行動変容測定の基準）

**目的**: ユーザーの初期状態と期待値を測定

**機能内容**:

プロフィール設定画面で自然に入力できる形式で実装（1-3と統合）

- 「猫を飼いたいと思っていますか？」
  - 非常に飼いたい / やや飼いたい / 迷っている / あまり飼いたくない
- 「飼育に対する不安はありますか？」（複数選択可）
  - 時間的負担 / 経済的負担 / 住環境 / 睡眠への影響 / 衛生面 / 生活の自由度 / その他
- 「どの程度、猫の飼育について調べましたか？」
  - 全く調べていない / 少し調べた / かなり調べた / 専門的に学んだ

**データ活用**:

- 終了時アンケートと比較し、行動変容を測定
- ユーザータイプ（A/B/Cタイプ）の判定

**成功基準**:

- 回答率: 90%以上
- 完答時間: 1分以内

**決定事項**:

- アンケート形式ではなく、プロフィール設定の一部として実装
- 必須項目を絞り、離脱率を抑える

---

### 1-2. ゲーム説明・期待値設定 🔴

**目的**: ユーザーに「気づき」を得るツールであることを明示

**機能内容**:

- オープニング画面で以下を伝える：
  - このゲームは「飼えるか/飼えないか」を判断するためのツール
  - 責任感を押し付けるのではなく、「気づき」を与える
  - 楽しさ + リアルな大変さを体験できる
  - 「やっぱりやめた」も成功として扱う
- 統計データの表示（最小限）
  - 令和5年度の猫の殺処分数: 6,899頭のみ表示
  - 詳細データは終了時レポート（4-2）で提示

**成功基準**:

- スキップ率: 30%未満（多くのユーザーが説明を読む）
- プレイ後アンケートで「目的を理解していた」: 80%以上

**決定事項**:

- オープニングでは殺処分数のみ表示（離脱率を抑える）
- 詳細な統計データは終了時レポート（4-2）で提示
- MVP期: テキストベース / 将来: グラフ化

---

### 1-3. 基本情報チェックリスト 🔴

**対応する理由**: 既に解決済み（調べれば済む）の領域

**目的**: A×X領域（基準が明確）の情報提供

**機能内容**:

- プロフィール設定画面で自然に入力できる形式で実装
- 経済的負担の確認
  - 自由に使えるお金/月（入力: 数値）
  - 初期費用2〜3万円の説明
- 時間的余裕の確認
  - 自由時間/日（入力: 数値）
  - 朝夕最低30分、トイレ掃除、ブラッシングの説明
- 住環境の確認
  - 住環境（選択肢: 持ち家ペット可 / 賃貸ペット可 / 賃貸ペット不可 / その他）
  - ペット可物件か？部屋の広さは十分か？
- バリデーション
  - 警告レベル: 基準未満でも警告を出してゲーム続行可能
  - エラーレベル: 賃貸ペット不可の場合は警告強め

**成功基準**:

- チェックリスト完了率: 70%以上

**決定事項**:

- プロフィール設定画面で自然に入力できる形式に変更
- チェックリスト形式ではなく、通常の入力フォームとして実装

## 2. コアゲームプレイ機能

### 2-1. 飼育シミュレーション（基本ループ） 🔴

**目的**: 日常のお世話を体験させる

**機能内容**:

- **1日のサイクル**（朝・昼・夕・夜の4フェーズ）
  - 朝: 食事準備、トイレ掃除
  - 昼: 仕事中（猫の様子を確認できない）
  - 夕: 食事準備、遊び、トイレ掃除
  - 夜: 睡眠（夜中に起こされるイベント）

- **基本的なお世話**
  - 食事やり（1日2回）
  - トイレ掃除（1日2〜3回）
  - 水の交換（1日1回）
  - 遊び（なつき度に影響）

- **状態管理**
  - 猫の健康度（0〜100）
  - 満腹度（0〜100）
  - なつき度（0〜100）
  - ストレス度（0〜100）
  - プレイヤーの体力（0〜100、睡眠で回復）

- **時間経過**
  - 1週間〜1ヶ月の体験（MVP範囲）
  - 時間スキップ機能（「1日進める」ボタン）

**成功基準**:

- シナリオ完了率: 60%以上
- 平均プレイ時間: 30分〜1時間

**決定事項**:

- MVP期はターン制（シナリオに応じた適切な時間単位）でシンプルに実装
- ゲーム性は最小限に抑え、体験の質を重視
- 育成ゲーム要素は避ける（楽しさよりリアルさを優先）

---

### 2-2. 最優先領域の体験イベント 🔴

**対応する理由（対応すべき理由の整理 No.1〜5, 15, 16）**:

- No.1: 睡眠不足・夜間の鳴き声（後悔頻出）
- No.2: 衛生面（後悔頻出）
- No.3: 日常の世話の継続性（20%）
- No.4: 生活の自由度の制限（26%）
- No.5: 家具・物への被害（後悔頻出）
- No.15: 近隣トラブル・苦情（3〜5%）
- No.16: 問題行動（1%）

**目的**: 「本当に我慢できる？」を疑似体験で問う

**機能内容**:

体験してみないとわからない日常の現実を、具体的なシナリオとして実装する。
詳細な実装方法（イベント内容、選択肢、影響など）は各シナリオ設計時に決定。

**成功基準**:

- 各イベントカテゴリを最低1回ずつ体験: 80%以上のユーザー
- 終了時アンケートで「印象に残ったイベント」に上位で挙がる

**決定事項**:

- イベント発生頻度はネガティブ:ポジティブ = 6:4（プレイテストで微調整）
- ゲームオーバーなし、状態が悪化するだけ（最後まで体験させる）
- ポジティブイベントを最低3個追加（2-5参照）

---

### 2-5. ポジティブな側面（喜び・癒し） 🔴

**目的**: 楽しさとリアルさのバランスを取る

**機能内容**:

猫との生活のポジティブな側面を体験できるイベントを実装。詳細は各シナリオ設計時に決定。

**決定事項**:

- 最低3個のポジティブイベントを実装
- イベント発生頻度: ネガティブ:ポジティブ = 6:4
- 工数: +6時間（各2時間 × 3イベント）

## 4. 振り返り・フィードバック機能

### 4-1. プレイログの記録 🔴

**目的**: ユーザーの行動を分析し、振り返りに活用

**機能内容**:

- 記録するデータ:
  - イベントごとの選択肢
  - 猫の状態変化（健康度、なつき度、ストレス度）
  - 総支出額
  - プレイ時間
  - 離脱ポイント（途中でやめた場合）

**データ活用**:

- 終了時レポート生成（4-2）
- プレイ後アンケートの質問カスタマイズ
- 開発側の分析（どのイベントで離脱が多いか）

---

### 4-2. 終了時レポート 🔴

**目的**: 気づきを振り返らせる

**機能内容**:

- **飼育適性診断**
  - あなたの飼育スタイル: 「マメなタイプ」「大雑把なタイプ」など
  - 苦手だったこと: 「睡眠不足」「衛生面」など
  - 得意だったこと: 「遊び相手」「スキンシップ」など

- **統計データとの比較**（詳細版）
  - 「あなたが体験したシーンは、実際の飼育放棄理由に基づいています」
  - 「睡眠不足で苦労したあなたは、実際の飼い主の43%と同じです」
  - 令和5年度の猫の殺処分数: 6,899頭
  - 主な原因: 飼い主の病気・介護（24%）、転居（8%）、問題行動（1%）

- **気づきポイントの明示**
  - 「こういう場面が苦手だった」リスト
  - 「実際はもっと大変」という注意書き
  - 「ゲームで飼えた ≠ 実際に飼える」という警告

**成功基準**:

- レポート閲覧率: 80%以上

**決定事項**:

- 詳細な統計データは終了時レポートで表示
- ユーザーが体験を振り返りながら統計データを参考に判断できる
- MVP期: テキストベース / 将来: グラフ化・インフォグラフィック化

---

### 4-3. プレイ後アンケート 🔴

**目的**: 行動変容を測定

**機能内容**:

- 「プレイ後、猫を飼いたいと思いますか？」
  - 非常に飼いたい / やや飼いたい / もう少し考えたい / やめておこうと思う
- 「プレイ前と比べて、飼育に対する理解は深まりましたか？」
  - 5段階評価
- 「どのシーンが最も印象に残りましたか？」
  - 自由記述
- 「このゲームは役に立ちましたか？」
  - 5段階評価
- 「文字情報だけでは気づけなかった点があったか？」
  - はい / いいえ / どちらともいえない

**データ活用**:

- 開始時アンケートと比較し、行動変容率を測定
- 印象に残ったシーンを分析し、体験設計を改善

**成功基準**:

- 回答率: 80%以上
- 行動変容率: 50%以上
- 「理解が深まった」: 80%以上

## 5. UI/UX機能

### 5-2. テキストベースの進行 🔴

**目的**: 開発コストを抑えつつ、体験の質を重視

**機能内容**:

- ノベルゲーム風のテキスト進行
  - イベント発生時にテキストで状況説明
  - 選択肢を提示
  - 結果をテキストで表示

**成功基準**:

- シナリオ完了率: 60%以上

**決定事項**:

- テキスト量のバランスはプレイテスト時に調整

## 6. マーケティング・データ分析機能

### 6-1. 統計データの表示 🔴

**目的**: 差別化ポイントをユーザーに伝える

**機能内容**:

- オープニング画面: 最小限の統計データ表示（1-2で言及済み）
  - 令和5年度の猫の殺処分数: 6,899頭のみ
- 終了時レポート: 詳細な統計データ表示（4-2で言及済み）
  - 「あなたの体験は実際の飼育放棄理由に基づいています」と明示
  - 主な原因の内訳、ユーザーの体験との比較

**成功基準**:

- プレイ後アンケートで「統計データに基づいた設計だと理解した」: 60%以上

**決定事項**:

- MVP必須に変更（MVP推奨から変更）
- オープニングでは最小限、終了時レポートで詳細表示
- 「ゲームクリアが存在しない」コンセプトに合致する配置

---

### 6-2. ユーザー行動ログの収集 🔴

**目的**: プロダクト改善のためのデータ収集

**機能内容**:

- プレイログ（4-1で詳細化済み）
- アンケート回答（1-1、4-3で詳細化済み）
- 離脱ポイント分析
- イベント別の印象度分析

**データ活用**:

- 体験設計の改善（どのイベントで離脱が多いか）
- 優先順位の再検証（対応すべき理由の整理.mdとの比較）

---

## 7. 収益化機能

### 7-1. 広告表示 🔴

**目的**: 初期収益化

**機能内容**:

- Google AdMob等の広告SDK統合
- 広告配置:
  - プレイ中の自然な区切り（1日の終わり）
  - 終了時レポート表示前
- 過度な広告は避ける（教育的価値を損なわない範囲）

**成功基準**:

- 広告が原因の離脱率: 10%未満

**決定事項**:

- 広告頻度はプレイテストで最適化（MVP期は1日の終わり + 終了時レポート前）
- 離脱率10%未満を維持することを優先
- 教育的価値を損なわない範囲で調整

## 8. 技術的機能

### 8-1. データ永続化 🔴

**目的**: プレイ進捗の保存とデータ分析

**機能内容**:

- クライアント側: ローカルストレージ（ブラウザ）でプレイ進捗を保存
- サーバー側: S3にプレイログを保存（データ分析用）
- セーブポイント: 1日の終わり

**決定事項**:

- クライアント側: ローカルストレージ（ブラウザ）で実装
- サーバー側: MVP期はS3への保存（ミニマム構成）
- プレイログをS3に保存することで、運営側でデータ分析が可能
- 将来バージョン: RDBやDynamoDBへの移行を検討

---

### 8-2. プラットフォーム対応 🔴

**目的**: ターゲットユーザーへのリーチ

**機能内容**:

- MVP期: Webアプリ（スマホブラウザ対応）
- 将来: iOS/Androidネイティブアプリ化

**決定事項**:

- MVP期はNext.js + Phaserで実装
- Webアプリとしてスマホブラウザ対応を優先
- 将来的にネイティブアプリ化が必要になった場合はReact Nativeへの移行を検討
