# ADR-0003: MVP範囲の定義（シナリオベース）

## ステータス

採用

## コンテキスト

機能一覧.mdでは「1週間〜1ヶ月の飼育体験」という期間ベースの記述があったが、実装の観点では**シナリオベース**で範囲を定義する方が適切である。

### 検討すべき点

- MVP開発期間: 1か月（104時間）
- 目標: シナリオ完了率60%、行動変容率50%
- 優先領域: A×Y領域（事前学習で防げる×基準が曖昧）に注力

### シナリオの定義

1つのシナリオ = 1つの完結したプレイ体験

- オンボーディング（アンケート、期待値設定）
- 飼育体験（基本ループ + イベント）
- 振り返り（レポート、アンケート）

## 決定

MVP範囲を以下のように定義する：

### MVP範囲: 最低1つの完結したシナリオを実装

- **シナリオ名**: 「1週間の飼育体験シナリオ」（仮称）
- **体験期間**: ゲーム内時間で1週間程度（プレイ時間30分〜1時間）
- **実装できるシナリオ数**: 1つ以上（時間に余裕があれば複数）

### 1つのシナリオに含まれる要素

#### 必須要素（絶対に実装）

1. **オンボーディング**
   - プレイ前アンケート（1-1）
   - ゲーム説明・期待値設定（1-2）

2. **コアゲームプレイ**
   - 飼育シミュレーション基本ループ（2-1）
     - 朝・昼・夕・夜の4フェーズ × 7日間
     - 基本的なお世話（食事、トイレ掃除、遊び）
   - A×Y領域の主要イベント（2-2）
     - 最優先: 問題行動のリアル（2-2-6）
     - 次点: 睡眠への影響（2-2-2）、衛生面の現実（2-2-3）
     - 最低3種類のイベントを実装
   - 計画外の繁殖イベント（2-4）

3. **金銭管理**
   - 飼育費用の可視化（3-1）
     - 初期費用、月次固定費、突発的費用の表示
     - 累積費用のグラフ

4. **振り返り**
   - プレイログの記録（4-1）
   - 終了時レポート（4-2）
   - プレイ後アンケート（4-3）

5. **UI/UX**
   - テキストベースの進行（5-2）
   - ビジュアルノベル型のUI

6. **技術**
   - データ永続化（8-1）
   - プラットフォーム対応（8-2: Webアプリ）

#### 推奨要素（実装したいが調整可能）

- 基本情報チェックリスト（1-3）
- 猫のビジュアル表現（5-1: 静止画 + シンプルなアニメーション）
- 統計データの表示（6-1）
- ユーザー行動ログの収集（6-2）
- 広告表示（7-1）

#### 将来バージョン（MVP後に実装）

- B×Y領域の老齢期イベント（2-3-2、2-3-3）
- A×Z領域のすべてのイベント（2-5）
- 1ヶ月体験シナリオ、多頭飼いシナリオ等

### 期間ではなくシナリオで区切る理由

1. **実装の柔軟性**
   - 「1週間」という期間に縛られず、必要なイベントを適切に配置できる
   - 実際には1週間より短い/長いかもしれないが、「1つの完結した体験」が重要

2. **品質の担保**
   - 「1週間のイベントを薄く実装」より「重要なイベントを濃く実装」する方が価値が高い
   - シナリオ完了率60%を達成するには、1つのシナリオの質が重要

3. **拡張性**
   - MVP後に「1ヶ月体験シナリオ」「多頭飼いシナリオ」等を追加しやすい
   - シナリオ単位でバリエーションを増やせる

## 影響

### メリット

- 実装範囲が明確（「1つのシナリオ」という単位）
- 品質重視の開発が可能（薄く広くより、濃く狭く）
- 将来の拡張が容易（シナリオ追加）

### デメリット

- 「1週間 vs 1ヶ月」という期間の議論が曖昧になる
  - 対策: 1つ目のシナリオは「1週間相当」の体験を目指すが、厳密には期間に縛られない

### 開発スケジュールへの影響

- Phase 1（2週間）: 1つ目のシナリオのコア部分を実装
- Phase 2（1週間）: イベント追加、体験の質向上
- Phase 3（1週間）: リリース準備

時間に余裕があれば、2つ目のシナリオ（より長期/異なるシチュエーション）を追加する。
