## 機能分類の方針

### 基本方針

- **[対応すべき理由の整理.md](41-対応すべき理由の整理.md)に基づいた優先順位づけ**
- **MVPは1週間～1ヶ月の飼育体験に絞る**
- **最優先領域（体験してみないとわからない × 後悔につながる）に注力**

### 優先度の定義

| 優先度 | アイコン | 説明 | 対応ドキュメント |
|---|---|---|---|
| **MVP必須** | 🔴 | 初期リリースで必ず実装する機能 | [51-MVP必須機能一覧.md](51-MVP必須機能一覧.md) |
| **MVP推奨** | 🟠 | 初期リリースで実装したい機能（工数次第で調整） | [52-MVP推奨機能一覧.md](52-MVP推奨機能一覧.md) |
| **将来バージョン** | 🟡 | 価値検証後に追加する機能 | [53-将来実装予定機能一覧.md](53-将来実装予定機能一覧.md) |
| **検討中** | ⚪ | 実装要否を検討する機能 | 各ドキュメントに分散 |

## 機能カテゴリ一覧

### 1. オンボーディング機能

| 機能 | 優先度 | 概要 |
|---|---|---|
| 1-1. プレイ前情報入力 | 🔴 MVP必須 | ユーザーの初期状態と期待値を測定 |

| 1-3. 基本情報チェックリスト | 🔴 MVP必須 | 経済・時間・住環境の確認 |

### 2. コアゲームプレイ機能

| 機能 | 優先度 | 概要 |
|---|---|---|

| 2-2. 最優先領域の体験イベント | 🔴 MVP必須 | 体験してみないとわからない日常の現実（No.1〜5, 15, 16） |

| 2-5. ポジティブな側面 | 🔴 MVP必須 | 楽しさとリアルさのバランス |
| 2-6. A×Z領域の体験イベント | 🟡 将来 | 猫を幸せにできるか |

### 3. 金銭管理機能

| 機能 | 優先度 | 概要 |
|---|---|---|

### 5. UI/UX機能

| No | 理由 | 対応機能 | 優先度 |
|---|---|---|---|
| 1 | 睡眠不足・夜間の鳴き声 | 2-2 | 🔴 MVP必須 |
| 2 | 衛生面 | 2-2 | 🔴 MVP必須 |
| 3 | 日常の世話の継続性 | 2-2 | 🔴 MVP必須 |
| 4 | 生活の自由度の制限 | 2-2 | 🔴 MVP必須 |

### 🟠 次点（未来×放棄）：全6項目をMVP推奨

| No | 理由 | 対応機能 | 優先度 |

| 8 | アレルギー発症 | 2-3-1 | 🟠 MVP推奨 |

### 🟡 補助的（知識×放棄）：2/5項目をMVP必須/推奨

## 開発スケジュール概算

**前提条件**:

- 開発可能時間: 平日2時間 × 20日 + 休日8時間 × 8日 = 104時間

**工数見積もり（合計104時間）**:

| Phase | カテゴリ | 工数 |
|---|---|---|
| **Phase 1** | 動くプロトタイプ | 60時間 |
| | 技術選定・設計 | 8時間 |
| | 基本ループ実装 | 16時間 |
| | 最優先イベント実装（主要2〜3種類） | 16時間 |
| | アンケート・レポート（簡易版） | 8時間 |
| | UI/UX実装（テキストベース） | 12時間 |
| **Phase 2** | 体験の質を上げる | 32時間 |
| | 最優先イベント追加 | 8時間 |
| | ポジティブイベント実装 | 6時間 |
| | プロフィール設定画面 | 4時間 |
| | 金銭管理機能 | 8時間 |
| | 終了時レポート充実 | 6時間 |
| **Phase 3** | リリース準備 | 12時間 |
| | 広告SDK統合 | 4時間 |
| | テスト・デバッグ | 8時間 |

→ **結論**: ギリギリ実現可能

## コンセプト達成の可否

### ✅ 網羅性

- **対応すべき理由の整理.mdの17項目をすべてカバー**
- 最優先領域（体験×後悔）に注力
- 次点領域（未来×放棄）も網羅的にカバー

### ✅ 行動変容の測定

- プレイ前・プレイ後アンケート
- プレイログ記録と終了時レポート
- 統計データとの比較

### ✅ 実装可能性

- 12月中の実装は可能（ただしギリギリ）
- Phase分けにより、段階的なリリースも可能
